Feature Selection¶

First I needed to determine which features, or columns to use in the training of the regression model. In other words, the end result of our machine learning model is a function $$f(z_0,z_1,z_2,...z_n) = volume_{hourly}$$ that accurately produces an hourly traffic volume given arguments $z_0...z_n$, and we need to choose which arguments $z$ train our model.

Among the columns present in the database, the only attribute that would be useful to predicting the hourly volume of traffic would be the timestamp of each car. However, the database contains individual records of vehicles, and since I wanted the hourly volume, I queried the database of vehicles that passed through the traffic station, grouping the cars into hourly bins, and chopped the timestamp into two pieces, the date and the time.

From the date I extracted several features, starting with the date itself, and more general features such as the day of week, week of the year, month, and season. From the time, we can glean the hour, whether the sun is up, and also whether its rush hour. My initial intuition for choosing from this pool was selecting the date, the hour, and the day of the week. The date, or day of the year, is the most specific denomination of the date section of the timestamp, and I felt that the day of the week was important to distinguish between weekdays and weekends, which would surely factor into traffic volumes. The hour of the day was a given inclusion. I did keep a mental note that these discrete features were as specific as the timestamp could produce, so if overfitting became an issue, then I would consider replacing the date with something more general, like week of the year. So to recap, the features I selected were the date, the hour, and day of the week.

Next I gave the type of variables some consideration. Hour of the day, date, and week day are all categorical variables. Regression models work with numerical variables, so this mismatch needed addressing. One solution I explored was one-hot encoding, which takes a categorical feature and creates a binary feature vector where each feature corresponds with one category. For example, day of the week is represented by a number $[0-6]$ (Sunday is 0), but after one-hot encoding, Sunday is $[1,0,0,0,0,0,0]$ and Monday is $[0,1,0,0,0,0,0]$.

However, the hour, day of the week, and date all possess an ordering of their discrete values. One-hot encoding is mainly for representing categorical features that have no inherent ordering, like red, white, and blue. Though it does make sense for Monday to be mapped to 1, Tuesday to 2, and so on. I explored both ways of representing my categorical features, and found that the model performed better with numerical encoding, over the one-hot encoding.

Early on, I compared the hour, day of week, date combination with other feature combinations on the AWS machine learning tool, and all the others performed worse, or had larger errors on average. Once I had figured out a good combination of features to represent the traffic volume function, I was done optimizing the AWS machine learning model, which would be incorporated alongside my Python regression model in an iOS visualization app.

To cater to each hour's traffic volume distribution, we treat them separately, determining independently their central tendency and standard deviation. As mentioned before, the arithmetic mean is not a good choice for centrality due to its sensitivity to extreme values (we have lots of extreme values). Therefore, the median is a far better statistic.

Z_SCORE = 3

dataframe.sort_values(by='HR',inplace=True)
for hour in range(0,24):
    data = dataframe.loc[dataframe.HR==hour]
    d = np.abs(data['TRAFFIC_COUNT'] - np.median(data['TRAFFIC_COUNT']))
    median_distance = np.median(d)
    absolute_distance = d/median_distance if median_distance else 0
    dataframe.loc[dataframe.HR==hour] = data[absolute_distance<Z_SCORE]
dataframe = dataframe.dropna(how='any')

In the above code snippet, I sort the whole dataset by hour and treat each hour as its own dataframe. The array data is a pandas dataframe that contains the rows of data where the feature HR == hour, and d contains the absolute value of the difference between each row's traffic volume and the median traffic volume. From there, I determine the median_distance from the median traffic volume, and scale the distance vector d by the median distance. The second to last line removes all rows whose traffic volume is too far from the median (> Z_SCORE * median_distance).

Regression¶

After cleaning the data, I started to construct the regression model. For this, I used the scikit-learn package, which provides an assortment of models and tools for preprocessing and cross validation. I paid close attention to underfitting and overfitting by choosing a ridge regression model and carefully tuning its parameters.

A ridge regression model is similar to a simple least squares regression model (LSRL), but the ridge model includes a regularization parameter. An LSRL is constructed by minimizing the error function $$\sum_{i=1}^{n} (y_i - \hat y_i)^2 $$ while the ridge regression minimizes $$\sum_{i=1}^{n} (y_i - \hat y_{i})^2 + \lambda \sum_{j = 1}^{m} \beta_j$$
where $y_i$ is the i-th observation, $\hat y_i$ is the corresponding generated i-th prediction, $n$ is the number of predictions generated, $m$ is the number of terms in the regression function, and $\beta_j$ is the coefficient that corresponds to the j-th feature, $x_j$.

It should be noted that $$ \hat y_i = \sum_{j=1}^{n}{x_{j} \beta_j}$$

The term added in the ridge regression error function limits the size of the coefficients in the regression function, with the degree of the limitation hinged on the size of $\lambda$. By imposing a limit on the coefficients, the model avoids overfitting the training data. This regularization parameter $\lambda$ is one of the parameters I tuned during cross-validation.

Another model parameter I fiddled with was the highest degree polynomial to use in my regression function. sci-kit learn actually implements this step in the preprocessing phase. It takes a training instance of the model, in our case [hour, day of week, date], and creates a new feature vector that contains all permutations of products of the features up to the degree specified. For example, for degree = 2, the feature vector would be $[hour^2, DOW^2, date^2, hour*DOW, DOW*date, hour*date, hour, DOW, date]$. From there, I would feed the feature vector into the model for training. Putting the determination of the polynomial degree in the preprocessing stage separated it from the cross-validation stage, which blocked me from autonomously trying each combination of degree polynomial and regularization parameter. I ended up manually trying combinations, which was not ideal.

I did run into an interesting issue when increasing the degree polynomial: when I decreased the regularization and increased the degree from 8 to 9, the error produced by the model dramatically increased, so much so that the coefficient of determination, or $R^2$ value, went from ~.90 to a negative value, meaning the regression model had become completely uncorrelated with the data. I investigated further and found that I might have run into my computer's threshold to succumbing to the curse of dimensionality (the feature vector for a 9th degree polynomial was over 100 features), or I encountered the result of an ill-conditioned design matrix, where a matrix used in the optimization process of the regression model becomes non-invertible, and the model cannot find coefficients from there forward.

The Deliverable¶

I ended up putting the regression model I made in a flask server and hosted the server on an AWS EC2 instance my manager had already set up for the project. The flask server acts as a simple API that responds to HTTP GET requests, and takes arguments from the URL. For example, http://www.wimnow.com:5001/success/0/223/3/2 grabs 2 days worth of predictions from the regression model starting from the 223rd day of the year (a Wednesday (corresponding to the 3)) at midnight (the 0).

I also wrote an iOS application that allows the user to input a date and amount of days of predictions, request the appropriate predictions from the server, and display them in a bar chart. I also allowed the user to specify whether to display predictions from the AWS machine learning model or the Python regression model.
